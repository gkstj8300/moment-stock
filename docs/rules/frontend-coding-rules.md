## 2. 중복 작성 금지

- 새 함수/컴포넌트를 만들기 전에 기존 코드에서 동일 기능이 있는지 반드시 검색한다.
- 검색 없이 새로 작성하지 않는다.

---

## 3. 요청 범위만 작업

- 사용자가 요청한 변경만 수행한다.
- 주변 코드 리팩토링, 주석 추가, 네이밍 변경 등 요청하지 않은 "개선"을 임의로 하지 않는다.

---

## 4. API/라이브러리 사용

- 라이브러리 API를 사용할 때 해당 API가 실제로 존재하는지 확인한다.
- 확신이 없으면 사용자에게 버전/API 존재 여부를 확인한다.
- 추측으로 코드를 작성하지 않는다.

---

## 5. 인터페이스 변경

- 파일 간 인터페이스(함수 시그니처, 타입, props 등)를 변경할 때 해당 인터페이스를 사용하는 모든 파일을 확인하고 함께 수정한다.
- 확인 없이 한쪽만 수정하지 않는다.

---

## 6. 근본 원인 해결 (땜질 금지)

- 에러/실패 발생 시 근본 원인을 먼저 파악한다. 원인 파악 없이 증상만 제거하는 수정을 하지 않는다.
- 타입 에러를 `as any`, `@ts-ignore`, `!`(non-null assertion) 등으로 우회하지 않는다.
- 테스트 실패 시 테스트 코드를 삭제하거나 `.skip`으로 우회하지 않는다. 실패 원인을 수정한다.
- 린트/빌드 에러를 `eslint-disable`, 설정 완화 등으로 무시하지 않는다.
- 원인 파악이 어려우면 사용자에게 상황을 보고하고 판단을 요청한다. 임의로 우회하지 않는다.

---

## 7. Import 규칙

### import type 분리

- 타입만 가져올 때 반드시 `import type { ... }` 구문을 사용한다.

### 경로 alias

- 프로젝트 내부 모듈은 `@/` alias를 사용한다.

---

## 8. Vanilla JS 제한

### 금지 API와 대체 수단

| 금지 API | 대체 수단 |
|----------|----------|
| `window.location.href` | `navigate()` (useNavigate 훅) |
| `window.location.pathname` | `location.pathname` (useLocation 훅) |
| `window.location.assign/replace()` | `navigate(url, { replace })` |
| `window.location.reload()` | 상태 초기화 + navigate; 불가피한 경우만 예외 |
| `document.querySelector` 등 | React `ref` (useRef) |

### 허용 예외

- `main.tsx`의 `document.getElementById('root')` — React 루트 마운트
- `localStorage` / `sessionStorage` — Zustand persist 등
- 예외 사용 시 `eslint-disable-next-line` + 사유 주석 필수

---

## 9. 디자인 리소스 임의 생성 금지

- 디자인과 관련된 모든 것(이미지, 아이콘, SVG, CSS 클래스, 색상값, 스타일 등)을 임의로 생성하지 않는다.
- 기존 프로젝트의 디자인 토큰, 컴포넌트, 에셋을 먼저 탐색하여 사용한다.
- 기존 리소스가 없으면 사용자에게 확인 후 제공받는다.

---

## 10. Spacing 규칙

- Figma auto layout에서 추출한 padding/gap 값을 기준으로 Tailwind 클래스를 적용한다.
- Tailwind v4 기본 spacing 스케일에 없는 값은 `index.css` `@theme`에 `--spacing-{n}` 토큰으로 등록한 뒤 사용한다.
- 임의의 arbitrary value(`[36px]` 등)는 사용하지 않는다 — 토큰으로 등록 후 사용한다.

---

## 11. 작업단위 가이드라인

### 안전 작업단위

1회 지시에서 누락 없이 완료할 수 있는 안전 범위:

| 작업 유형 | 안전 범위 |
|-----------|-----------|
| 기능 구현 | 1기능 (1페이지 또는 1컴포넌트 단위) |
| 리팩토링 | 관련 파일 5개 이내 |
| 버그 수정 | 1이슈 |

### 위험 작업단위

아래 조건에 해당하면 반드시 분할하여 진행한다.

| 기준 | 예시 |
|------|------|
| 파일 10개 이상 동시 변경 | "모든 페이지에 로딩 스피너 추가해" |
| 범위가 열린 작업 | "전부 수정해", "모든 파일에 적용해" |
| 여러 feature 영역에 걸친 교차 변경 | "인증 + 근태 + 관리자 동시에 API 연결해" |

분할 방법: 변경 대상 파일 나열 → 기능 영역/파일 수 기준으로 단계 분리 → 각 단계를 별도 지시로 요청.

## 12. 지시 형식 권장사항

### 여러 이슈를 한 번에 처리할 때

번호로 명시적으로 나열한다: `C4, M1, M4 수정해`

### 범위를 특정할 수 있는 구체적 지시어 사용

| 비권장 | 권장 |
|--------|------|
| "다 고쳐" | "LoginPage의 폼 유효성 검사 수정해" |
| "전체적으로 리팩토링해" | "attendance 폴더의 타입 정의를 types.ts로 분리해" |
| "모든 파일에 적용해" | "src/pages/ 하위 6개 페이지에 ErrorBoundary 적용해" |

### 대규모 작업의 단계별 분할

파일 10개 이상 또는 여러 영역에 걸치는 작업은 단계별로 분할한다.

## 13. 프로젝트 실전 교훈

### 실패 사례

| 문제 | 원인 | 결과 |
|------|------|------|
| 기능명세서 항목 다수 누락 | 원본 명세서(CSV)를 대조하지 않고 기억에 의존 | 직급, 휴직/입사대기 상태 등 누락 |
| 워크플로우 단계 건너뛰기 | 짧은 응답("ㅇㅇ")을 go 사인으로 오해석 | Figma 참조 없이 구현 → 품질 저하 |
| 메모리 갱신 누락 | 구현 완료 후 MEMORY.md 업데이트를 생략 | 다음 세션에서 중복 작업 |
| Calendar hover 중첩 수정 방향 오류 | 부모/자식 hover 소유권 판단 실패 | 롤백 필요 |
| 라이브러리 export명 오기억 | `Building` → 실제는 `Buildings` | 사용자 지적 후 수정 |

### 교훈 요약

1. **원본 대조 필수**: 기능명세서·API 요청서 등 기반 문서를 반드시 먼저 읽는다.
2. **단계 건너뛰기 금지**: 워크플로우 6단계를 반드시 순서대로 따른다.
3. **메모리 즉시 갱신**: 작업 완료 직후 MEMORY.md를 업데이트한다.
4. **수정 전 실코드 확인 필수**: 문서/추론만으로 판단하지 말고 실제 코드를 읽어서 현 상태를 확인한 뒤 수정한다.
5. **부모-자식 스타일 소유권 판단**: hover/active 등 상호작용 스타일이 중첩될 때, 범용 컴포넌트(ui/)가 소유하고 사용처(features/)가 위임받는 구조가 기본이다.
6. **라이브러리 export명 확인**: 아이콘·컴포넌트 이름을 기억에 의존하지 말고 패키지 검색으로 정확한 이름을 확인한다.
