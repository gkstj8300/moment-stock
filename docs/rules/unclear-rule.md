# 모호한 지시 처리 규칙
사용자의 지시가 검증 불가능한(모호한) 표현을 포함하면, 임의로 해석하여 작업하지 않는다. 반드시 구체적인 기준을 사용자에게 되물어야 한다.

## 검증 가능한 지시란?
Yes/No로 준수 여부를 판단할 수 있는 지시. 수치, 패턴, 조건 중 하나 이상 포함.

## 좋은 규칙의 3요소
1. **수치가 있다** — 줄 수, 개수, 횟수 등 숫자로 표현
2. **패턴이 있다** — camelCase, try-catch, 파일 위치 등 형태로 표현
3. **조건이 있다** — "~하면 ~한다"처럼 트리거와 액션이 명확

## 되묻기 예시

| 사용자 지시 | 판정 | 에이전트 행동 |
|---|---|---|
| "깔끔하게 작성해" | 모호 | "깔끔하다의 기준이 무엇인가요? (예: 파일당 200줄 이내, 함수당 30줄 이내 등)" |
| "적절히 에러 처리해" | 모호 | "어떤 범위에 어떤 방식으로 처리할까요? (예: 외부 API 호출은 try-catch 필수 등)" |
| "파일당 200줄 이내로 작성해" | 검증 가능 | 바로 작업 진행 |
| "좋은 구조로 만들어" | 모호 | "좋은 구조의 기준이 무엇인가요? (예: 파일당 단일 책임, 3depth 이내 등)" |
| "컴포넌트 3개로 분리해" | 검증 가능 | 바로 작업 진행 |
| "성능 좋게 해줘" | 모호 | "어떤 성능 기준인가요? (예: LCP 2.5s 이내, 번들 200KB 이내 등)" |
| "이미지는 next/image로 렌더링해" | 검증 가능 | 바로 작업 진행 |

## 질의응답 기록

모호한 지시에 대해 되물은 후, 사용자의 응답을 기록으로 남긴다.
다음에 같은 종류의 지시가 왔을 때 참고하고, 사용자가 자신의 판단 패턴을 학습할 수 있게 한다.

### 기록 위치
- 폴더: `{front|back}/docs/logs-clarifications/`
- 파일명: `YYYY-MM-DD_주제.md` (예: `2026-03-06_에러처리-기준.md`)
- 같은 날 같은 주제면 기존 파일에 추가. 주제가 다르면 새 파일 생성.

### 기록 형식

```markdown
# 명확화 기록: (주제)

## 날짜
YYYY-MM-DD

## 원래 지시
(사용자의 원래 표현)

## 모호한 부분
(어떤 부분이 검증 불가능했는지)

## 에이전트 질문
(되물은 내용)

## 사용자 응답
(사용자가 구체화한 기준)

## 확정된 규칙
(최종적으로 적용한 검증 가능한 규칙)
```

### 기록 활용
- 동일하거나 유사한 모호 지시가 다시 올 경우, 기존 기록을 참고하여 "이전에 이렇게 정하셨는데, 이번에도 동일한 기준을 적용할까요?"로 되물을 수 있다.
- 단, 이전 기록을 근거로 임의 적용하는 것은 금지. 반드시 확인한다.
