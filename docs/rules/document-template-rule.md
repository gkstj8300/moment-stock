# 문서 작성 규격 & 템플릿

이 프로젝트의 **모든 마크다운 문서**(`*.md`)는 아래 규격을 따른다. 프론트엔드, 백엔드, 기타 영역 구분 없이 동일하게 적용한다.

---

## 템플릿

새 문서를 작성할 때 아래를 복사하여 사용한다. 필드를 생략하거나 순서를 변경하지 않는다.

```markdown
# {문서 제목}

## 1. 문서 정보

| 항목 | 내용 |
|------|------|
| 문서명 | {문서의 정식 명칭} |
| 버전 | v0.1.0 |
| 작성일 | {YYYY-MM-DD} |
| 기반 문서 | {프로젝트 루트 기준 경로 + 확장자, 쉼표 구분. 없으면 "없음"} |

### 변경 이력

| 버전 | 날짜 | 작성자 | 변경 내용 |
|------|------|--------|-----------|
| v0.1.0 | {YYYY-MM-DD} | {작성자} | 신규 작성 |

---

## 2. {첫 번째 섹션 제목}
```

---

## 규칙 상세

1. **문서 정보 테이블**: 4개 필드(문서명, 버전, 작성일, 기반 문서) 고정. 추가 필드 불가.
2. **기반 문서**: 프로젝트 루트 기준 상대 경로와 확장자를 반드시 포함한다. 형식 제한 없음(`.md`, `.csv`, `.xlsx`, `.docx` 등).
   - 올바른 예: `/docs/spec/example_v1.0.csv`, `/backend/docs/db-schema.md`, `/docs/설계서.docx`
   - 잘못된 예: `기능명세서`, `example_v1.0`
3. **버전**: Semantic Versioning(v메이저.마이너.패치)을 사용한다.
4. **변경 이력 테이블**: 4개 컬럼(버전, 날짜, 작성자, 변경 내용) 고정. 모든 수정 시 행을 추가한다. 기존 행은 삭제하지 않는다.
5. **변경 내용 작성법**: 어떤 항목이 어떻게 바뀌었는지 구체적으로 기술한다.
   - 올바른 예: `로그인 API 응답에 refreshToken 필드 추가, 에러 코드 체계 정의`
   - 잘못된 예: `내용 수정`, `업데이트`

## 문서 수정 체크리스트

프로젝트 내 문서를 수정할 때 반드시 아래 절차를 따른다.

### 1. 수정 전: 원본 대조

- 수정 대상 문서의 **기반 문서** 항목에 명시된 파일을 **반드시 먼저 읽는다**
- 기반 문서의 경로나 파일명이 바뀌었을 수 있으므로, 해당 경로에 파일이 없으면 디렉터리를 확인하여 최신 파일을 찾는다

### 2. 수정 시: 기반 문서 대비 검증

수정 내용을 반영한 후, 기반 문서와 대조하여 아래를 확인한다:

- [ ] **내용 완전성**: 기반 문서의 모든 항목과 세부 사항이 작성 문서에 빠짐없이 반영되었는지 확인
- [ ] **부가 정보 반영**: 기반 문서의 비고, 주석, 조건 등 부가 정보가 문서에 반영되었거나 `[TBD]`로 명시되었는지 확인

### 3. 수정 후: 메타 정보 갱신

- [ ] **변경 이력 추가**: 문서 상단 변경 이력 테이블에 새 행 추가 (버전, 날짜, 작성자, 변경 내용)
- [ ] **기반 문서 경로 확인**: 기반 문서가 변경된 경우(파일명, 확장자, 경로) 문서 정보 테이블 갱신
- [ ] **문서 간 정합성**: 동일한 정보를 참조하는 다른 문서와 일치 여부 확인
