## 2. 프론트엔드 개요

### 2.1 플랫폼 구성

moment-stock의 프론트엔드는 두 개의 플랫폼으로 구성된다.

| 플랫폼 | 기술 | 주요 역할 |
|--------|------|-----------|
| **웹(Web)** | Next.js 14 (App Router) | SEO 확보, ISR 기반 고속 상품 페이지 서빙, 실시간 재고 스트리밍 |
| **앱(App)** | React Native Expo | 푸시 알림, 생체 인증 간편 결제, 오프라인 복원력 |

두 플랫폼은 TurboRepo 기반 모노레포 내 `@repo/shared` 패키지를 통해 비즈니스 로직, 타입, 유틸리티를 공유한다.

### 2.2 핵심 UX 목표

타임 세일의 **긴박감(urgency)** 을 시각적으로 전달하면서도 부드럽고 반응성 높은 인터페이스를 유지하는 것이 프론트엔드의 최우선 목표이다.

- 재고 수량 변화를 100ms 이내에 화면에 반영
- 고빈도 업데이트 상황에서도 60fps 이상의 렌더링 성능 유지
- 트래픽 폭주 시 대기열(Waitlist Queue)을 통한 안정적인 사용자 경험 제공

## 3. 웹 아키텍처 (Next.js 14)

### 3.1 App Router 구조

Next.js 14의 App Router를 채택하여 서버 컴포넌트와 클라이언트 컴포넌트를 명확히 분리한다.

- **서버 컴포넌트**: 상품 정보, 카테고리 목록 등 정적 데이터를 ISR로 서빙
- **클라이언트 컴포넌트**: 재고 수량 표시, 장바구니 인터랙션, 대기열 UI 등 실시간성이 필요한 영역

### 3.2 렌더링 전략

| 전략 | 적용 대상 | 설명 |
|------|-----------|------|
| **ISR** | 상품 상세 페이지, 카테고리 페이지 | 상품 기본 정보(이미지, 설명, 가격)를 빌드 시 생성하고 주기적으로 재검증 |
| **클라이언트 실시간** | 재고 수량, 타임 세일 카운트다운 | Supabase Realtime 구독을 통해 클라이언트에서 실시간 렌더링 |
| **SSR** | 주문 내역, 마이페이지 | 사용자별 데이터를 요청 시점에 서버에서 렌더링 |

### 3.3 라우트 구조 개요

```
app/
├── (public)/              # 비인증 영역
│   ├── page.tsx           # 홈
│   ├── products/
│   │   ├── page.tsx       # 상품 목록
│   │   └── [id]/
│   │       └── page.tsx   # 상품 상세 (ISR)
│   └── time-sale/
│       └── page.tsx       # 타임 세일 랜딩
├── (auth)/                # 인증 필요 영역
│   ├── cart/
│   │   └── page.tsx       # 장바구니
│   ├── checkout/
│   │   └── page.tsx       # 결제
│   └── orders/
│       └── page.tsx       # 주문 내역
└── api/                   # API 라우트
```

### 3.4 미들웨어 고려사항

- **인증 미들웨어**: Supabase Auth 세션 검증, 미인증 사용자의 보호 경로 접근 시 로그인 페이지로 리다이렉트
- **대기열 미들웨어**: 타임 세일 시간대에 트래픽 폭주 감지 시 대기열 페이지로 분기
- **지역/언어 미들웨어**: 향후 다국어 지원을 위한 확장 포인트 확보

## 4. 앱 아키텍처 (React Native Expo)

### 4.1 네비게이션 구조

Expo Router를 활용한 파일 기반 네비게이션을 채택한다.

```
app/
├── (tabs)/                # 탭 네비게이션
│   ├── index.tsx          # 홈
│   ├── products.tsx       # 상품 목록
│   ├── cart.tsx           # 장바구니
│   └── mypage.tsx         # 마이페이지
├── product/[id].tsx       # 상품 상세
├── checkout.tsx           # 결제
├── orders.tsx             # 주문 내역
└── time-sale.tsx          # 타임 세일
```

### 4.2 푸시 알림 연동 (FCM)

- **트리거 조건**: 관심 상품 재고 임박(예: 5개 미만), 타임 세일 시작 직전 알림
- **구현 흐름**: Supabase Edge Functions에서 재고 임계값 도달 감지 → FCM을 통해 디바이스로 푸시 전송
- **사용자 설정**: 알림 수신 여부 및 임계값을 사용자별로 설정 가능

### 4.3 생체 인증

- Expo LocalAuthentication API를 활용한 FaceID/지문 인증
- 결제 확인, 주문 승인 등 민감한 작업 수행 전 생체 인증 필수 적용
- 생체 인증 미지원 디바이스에서는 PIN 입력으로 폴백(fallback)

### 4.4 오프라인 모드 처리

| 단계 | 동작 |
|------|------|
| **오프라인 진입** | 네트워크 단절 감지 시 로컬 캐시 데이터 기반으로 UI 유지, 오프라인 배너 표시 |
| **오프라인 중** | 장바구니 편집 등 로컬 작업은 큐에 저장, 결제 등 서버 의존 작업은 차단 |
| **복구 시** | 네트워크 복원 감지 → Supabase Realtime 재구독 → 최신 재고 상태 재동기화(Re-sync) → 로컬 큐 작업 순차 전송 |

## 5. 공유 계층 설계

### 5.1 공유 대상 (`@repo/shared`)

| 분류 | 항목 | 설명 |
|------|------|------|
| **타입** | 상품, 주문, 사용자 등 도메인 타입 | Zod 스키마와 TypeScript 타입을 함께 관리 |
| **훅** | `useRealtimeStock`, `useTimeSaleCountdown` 등 | Supabase Realtime 구독 및 TanStack Query 래퍼 훅 |
| **유틸리티** | 가격 포맷팅, 재고 상태 판별, 날짜 계산 | 플랫폼 무관한 순수 함수 |
| **Supabase 클라이언트** | 클라이언트 초기화, RPC 호출 래퍼 | 환경변수만 플랫폼별로 주입 |
| **상수** | 재고 임계값, 타임 세일 상태 코드, 에러 코드 | 비즈니스 규칙 관련 상수 |

### 5.2 플랫폼 전용 영역

| 영역 | 웹 (Next.js) | 앱 (React Native Expo) |
|------|-------------|----------------------|
| **UI 컴포넌트** | Tailwind CSS 기반 컴포넌트 | Tamagui 기반 컴포넌트 |
| **네비게이션** | App Router (파일 기반 라우팅) | Expo Router (스택/탭 네비게이션) |
| **인증 보조** | 쿠키 기반 세션 | 생체 인증, SecureStore 토큰 저장 |
| **알림** | 브라우저 Notification API (선택적) | FCM 푸시 알림 |
| **렌더링** | ISR, SSR, CSR 혼합 | 클라이언트 전용 렌더링 |
| **오프라인** | Service Worker (선택적) | NetInfo + AsyncStorage 기반 큐 |

## 6. 렌더링 최적화 전략

### 6.1 React.memo를 활용한 재고 표시 컴포넌트 최적화

재고 수량 표시 컴포넌트는 빈번하게 업데이트되므로 `React.memo`로 감싸 불필요한 리렌더링을 방지한다. 상품 카드 내에서 재고 수량 영역만 독립 컴포넌트로 분리하여 해당 부분만 갱신되도록 설계한다.

### 6.2 원자적 상태 관리(Atomic State)

Zustand의 선택적 구독(selector) 패턴을 활용하여 상태 변경의 영향 범위를 최소화한다. 재고 수량, 타임 세일 잔여 시간 등 고빈도 변경 데이터는 별도의 원자 단위 스토어로 분리한다.

### 6.3 requestAnimationFrame 기반 숫자 전환

재고 수량이 변경될 때 `requestAnimationFrame`을 사용하여 숫자가 부드럽게 전환되도록 애니메이션을 적용한다. 이를 통해 60fps를 유지하면서도 시각적 긴박감을 전달한다.

### 6.4 가상화 리스트(Virtualized List)

상품 목록 페이지에서는 가상화 리스트를 적용하여 화면에 보이는 항목만 렌더링한다.

- **웹**: `react-window` 또는 `@tanstack/react-virtual` 활용
- **앱**: React Native의 `FlashList` 활용

## 7. 페르소나 관점 토론

### Monorepo Lead 의견

> "코드 공유의 경계를 명확히 설정하는 것이 모노레포 성공의 핵심이다."

- **공유 디렉터리 전략**: `@repo/shared`에는 플랫폼 독립적인 순수 로직만 포함한다. UI 관련 코드는 절대 공유 패키지에 포함하지 않는다.
- **플랫폼별 디렉터리**: `apps/web`과 `apps/mobile`은 각각 독립적으로 빌드 가능해야 하며, 공유 패키지에 대한 의존성만 허용한다.
- **빌드 최적화**: TurboRepo의 캐싱 전략을 활용하여 변경된 패키지만 재빌드한다. 공유 패키지 변경 시 양쪽 플랫폼의 빌드가 트리거되므로, 공유 패키지의 변경 빈도를 최소화하는 인터페이스 설계가 중요하다.
- **의존성 관리**: 공유 패키지의 외부 의존성은 최소한으로 유지한다. 플랫폼별 polyfill이 필요한 라이브러리는 공유 계층에서 제외한다.

### Engine Designer 의견

> "프론트엔드의 API 계층은 Supabase RPC의 제약 조건을 정확히 반영해야 한다."

- **API 계층 설계**: Supabase RPC 호출을 직접 사용하지 않고, 도메인별 서비스 레이어를 두어 추상화한다. 이를 통해 RPC 시그니처 변경 시 영향 범위를 제한한다.
- **에러 핸들링**: DB 제약 조건(재고 부족, 중복 주문 등)에서 발생하는 에러를 프론트엔드에서 사용자 친화적인 메시지로 변환하는 에러 매핑 계층이 필요하다.
- **Optimistic Update 설계**: 재고 차감 시 낙관적 업데이트를 적용하되, Supabase RPC의 원자적 검증 결과에 따라 롤백을 수행한다. TanStack Query의 `onMutate`/`onError` 훅을 활용한다.
- **Edge Function 연동**: 결제 검증, 재고 임계값 알림 등 보안이 필요한 로직은 Edge Function을 통해 처리하며, 프론트엔드는 해당 엔드포인트만 호출한다.

### Sync Master 의견

> "실시간 구독의 생명주기 관리가 두 플랫폼 모두에서 안정적이어야 한다."

- **공유 구독 훅**: `useRealtimeStock` 훅은 `@repo/shared`에 위치하되, 구독 채널 생성/해제 로직을 플랫폼 독립적으로 구현한다. 컴포넌트 마운트/언마운트 시 구독이 확실히 정리되어야 한다.
- **TanStack Query 설정 차이**: 웹에서는 탭 포커스 시 자동 리패치(`refetchOnWindowFocus`)를 활성화하고, 앱에서는 앱 포그라운드 복귀 시 리패치를 트리거한다. `staleTime`과 `gcTime`은 플랫폼별로 조정 가능하도록 설정 파일을 분리한다.
- **캐시 전략 차이**: 웹은 메모리 캐시 위주로 운영하고, 앱은 AsyncStorage를 활용한 영속 캐시를 추가하여 오프라인 복원력을 확보한다.
- **재연결 전략**: 네트워크 단절 후 복구 시 마지막 수신 이벤트 이후의 변경분을 보정하는 재동기화 로직을 공유 훅 수준에서 제공한다.

### UI Artisan 의견

> "타임 세일 플랫폼의 UI는 긴박감과 신뢰감을 동시에 전달해야 한다. 이것이 이 프로젝트의 프론트엔드가 풀어야 할 가장 핵심적인 과제이다."

#### 디자인 시스템 전략 (Tailwind vs Tamagui)

웹은 Tailwind CSS, 앱은 Tamagui를 사용하되, **디자인 토큰(색상, 간격, 타이포그래피, 그림자 등)은 공유 패키지에서 단일 소스로 관리**한다. 각 플랫폼의 스타일링 도구가 이 토큰을 소비하는 구조를 취하여 시각적 일관성을 보장한다.

- `@repo/shared/design-tokens`: 색상 팔레트, 간격 스케일, 폰트 크기 등을 JSON 또는 TypeScript 상수로 정의
- 웹: `tailwind.config.ts`에서 토큰을 import하여 Tailwind 테마로 변환
- 앱: `tamagui.config.ts`에서 토큰을 import하여 Tamagui 테마로 변환

#### 컴포넌트 아키텍처

플랫폼별 UI 컴포넌트는 독립적으로 구현하되, **컴포넌트 인터페이스(Props 타입)는 공유**하여 동일한 데이터 모델에서 동작하도록 한다. 이를 통해 비즈니스 로직 훅이 양쪽 플랫폼의 컴포넌트에 동일하게 연결될 수 있다.

#### 시각적 긴박감 패턴

타임 세일의 핵심 감정인 긴박감을 전달하기 위해 다음 UI 패턴을 적용한다.

| 패턴 | 설명 | 적용 시점 |
|------|------|-----------|
| **카운트다운 타이머** | 남은 시간을 시:분:초로 표시, 1분 미만 시 붉은색 강조 | 타임 세일 진행 중 |
| **재고 게이지 바** | 잔여 재고를 프로그레스 바로 시각화, 잔량 감소 시 색상 변화(초록→노랑→빨강) | 상품 상세, 상품 카드 |
| **숫자 롤링 애니메이션** | 재고 수량 변경 시 슬롯머신 방식의 숫자 전환 효과 | 실시간 재고 표시 영역 |
| **진동 피드백** | 재고 5개 미만 시 "거의 품절" 배지 진동 애니메이션 | 상품 카드, 상품 상세 |
| **희소성 배지** | "마지막 3개!", "곧 품절" 등 텍스트 배지 | 상품 카드 |

#### 접근성(Accessibility)

- 모든 인터랙티브 요소에 적절한 ARIA 레이블 제공
- 카운트다운 타이머는 `aria-live="polite"`로 스크린 리더에 주기적 알림
- 색상만으로 정보를 전달하지 않고, 텍스트 또는 아이콘을 병행
- 터치 타겟은 최소 44x44px 확보
- 애니메이션 과민 사용자를 위한 `prefers-reduced-motion` 미디어 쿼리 대응

#### 반응형 디자인

- 웹: 모바일(360px) → 태블릿(768px) → 데스크톱(1280px) 3단계 브레이크포인트
- 상품 그리드: 모바일 1열 → 태블릿 2열 → 데스크톱 3~4열
- 타임 세일 배너: 모바일에서는 고정 상단 바, 데스크톱에서는 히어로 섹션

#### 애니메이션 전략

- **웹**: CSS transitions/animations 우선, 복잡한 인터랙션은 Framer Motion 활용
- **앱**: React Native Reanimated를 활용한 네이티브 드라이버 기반 애니메이션
- **공통 원칙**: 200ms~300ms 지속 시간, ease-out 곡선을 기본값으로 사용하여 자연스러운 느낌 유지

## 8. 주요 페이지/화면 구성

| 페이지/화면 | 설명 | 렌더링 방식 |
|------------|------|-------------|
| **홈** | 진행 중인 타임 세일 배너, 인기 상품, 추천 상품 노출 | 웹: ISR + 클라이언트 실시간(타임 세일 영역) / 앱: CSR |

| **상품 상세** | 상품 이미지, 설명, 가격 등 정적 정보 + 실시간 재고 수량 | 웹: ISR(정적 정보) + CSR(재고 실시간) / 앱: CSR |
| **장바구니** | 담은 상품 목록, 수량 변경, 재고 초과 시 경고 표시 | 웹: CSR / 앱: CSR + 로컬 큐 |

| **주문 내역** | 과거 주문 목록 및 상태 조회 | 웹: SSR / 앱: CSR + 영속 캐시 |
| **타임 세일 랜딩** | 예정/진행 중인 타임 세일 목록, 카운트다운, 대기열 진입점 | 웹: ISR + CSR(카운트다운, 대기열) / 앱: CSR + 푸시 알림 연동 |

## 9. 대기열 시스템 (Waitlist Queue)

### 9.1 개요

타임 세일 시작 시 발생하는 트래픽 폭주로부터 시스템을 보호하고, 동시에 사용자에게 공정한 접근 순서를 제공하기 위한 클라이언트 사이드 가상 대기열 시스템이다.

### 9.2 동작 흐름

1. **진입 판정**: 타임 세일 시간대에 접속량이 임계값을 초과하면 미들웨어(웹) 또는 앱 로직에서 대기열 페이지/화면으로 분기
2. **대기열 등록**: 서버에 대기열 등록 요청 → 고유 대기 토큰 및 예상 순번 수신
3. **순번 표시**: 실시간 채널을 통해 현재 대기 순번을 주기적으로 갱신하여 사용자에게 표시
4. **입장 허가**: 순번이 도달하면 서버로부터 입장 토큰 수신 → 타임 세일 페이지로 자동 리다이렉트

### 9.3 대기열 UX

| 요소 | 설명 |
|------|------|
| **현재 순번 표시** | "현재 대기 순번: 142번 / 예상 대기 시간: 약 3분" 형태로 실시간 갱신 |
| **프로그레스 표시** | 전체 대기열 대비 현재 위치를 프로그레스 바로 시각화 |
| **이탈 방지** | 페이지/앱을 떠나면 순번이 취소될 수 있음을 안내하는 경고 표시 |
| **알림 옵션** | 앱에서는 순번 도달 시 푸시 알림 수신 옵션 제공 |
| **대기 중 콘텐츠** | 대기 중 타임 세일 상품 미리보기, 추천 상품 등을 노출하여 이탈률 감소 |
