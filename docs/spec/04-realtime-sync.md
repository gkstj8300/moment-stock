## 2. 실시간 동기화 개요

### 왜 실시간 동기화가 필요한가

moment-stock은 타임 세일 환경에서 운영되는 커머스 플랫폼이다. 타임 세일은 다음과 같은 특성을 갖는다.

- **한정된 시간**: 세일 시작과 동시에 수백~수천 명의 사용자가 동시 접속한다.
- **한정된 수량**: 재고가 초 단위로 소진되며, 사용자는 남은 수량을 실시간으로 확인해야 한다.
- **공정성 요구**: 화면에 표시된 재고와 실제 재고의 불일치는 사용자 신뢰를 훼손한다.

이러한 환경에서 재고 상태의 지연 전파는 초과 판매(Overselling), 사용자 이탈, 클레임 증가로 직결된다.

### 목표 지표

| 지표 | 목표 값 |
|------|---------|
| DB 변경 → 클라이언트 반영 지연 | 100ms 이내 |
| UI 프레임 유지 | 60fps 이상 |
| 네트워크 복구 후 재동기화 | 자동 수행 |
| 초과 판매 발생률 | 0% |

## 3. 동기화 아키텍처

### Supabase Realtime CDC 파이프라인

Supabase Realtime은 PostgreSQL의 논리적 복제(Logical Replication)를 기반으로 한 CDC(Change Data Capture) 메커니즘을 제공한다. DB 테이블의 변경 사항이 발생하면, Realtime 서버가 이를 감지하여 WebSocket으로 연결된 클라이언트에 브로드캐스트한다.

### 전체 흐름도

```
┌─────────────────────────────────────────────────────────────────────┐
│                        PostgreSQL (Supabase)                       │
│                                                                     │
│  [사용자 A 구매] → UPDATE stock SET quantity = quantity - 1         │
│        │                                                            │
│        ▼                                                            │
│  WAL (Write-Ahead Log) 기록                                        │
│        │                                                            │
│        ▼                                                            │
│  supabase_realtime Publication 감지                                 │
└────────┬────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────┐
│  Supabase Realtime 서버  │
│  (CDC 이벤트 수신)       │
│        │                 │
│        ▼                 │
│  채널별 브로드캐스트      │
└────────┬────────────────┘
         │ WebSocket
         ▼
┌─────────────────────────────────────────────────────┐
│                    클라이언트                         │
│                                                      │
│  WebSocket 수신                                      │
│        │                                             │
│        ▼                                             │
│  이벤트 스로틀링 / 배칭                               │
│        │                                             │
│        ▼                                             │
│  TanStack Query 캐시 업데이트                         │
│        │                                             │
│        ▼                                             │
│  React 리렌더링 (requestAnimationFrame)              │
│        │                                             │
│        ▼                                             │
│  사용자 화면 반영                                     │
└──────────────────────────────────────────────────────┘
```

### 채널 구독 구조

| 채널 | 구독 대상 | 이벤트 유형 |
|------|-----------|-------------|
| `product-stock:{productId}` | 개별 상품 재고 변경 | UPDATE |
| `time-sale:{saleId}` | 타임 세일 상태 변경 (시작, 종료, 연장) | UPDATE, INSERT |
| `order-status:{userId}` | 사용자 본인 주문 상태 변경 | UPDATE |

## 4. 클라이언트 상태 관리 전략

### TanStack Query v5 — 서버 상태 캐싱

TanStack Query는 서버에서 가져온 데이터의 캐싱, 동기화, 갱신을 담당한다. 실시간 구독과 결합하여 서버 상태의 단일 진실 공급원(Single Source of Truth)을 유지한다.

| 역할 | 설명 |
|------|------|
| 쿼리 키 설계 | `['product', productId, 'stock']` 형태로 재고 데이터를 세분화 |
| 캐시 무효화 | Realtime 이벤트 수신 시 `queryClient.setQueryData`로 직접 캐시 갱신 |
| Stale 전략 | 재고 데이터는 `staleTime: 0`으로 설정하여 항상 최신 상태 유지 |
| 재시도 | 네트워크 오류 시 지수 백오프(Exponential Backoff)로 자동 재시도 |

### Zustand — 클라이언트 전용 전역 상태

Zustand는 서버와 무관한 순수 클라이언트 측 상태를 관리한다.

| 상태 | 설명 |
|------|------|
| 장바구니(Cart) | 사용자가 담은 상품 목록 (서버 전송 전 로컬 상태) |
| 모달/토스트 | UI 표시 여부 제어 |
| 필터/정렬 | 상품 목록 필터링 조건 |
| 구독 연결 상태 | WebSocket 연결 여부 (온라인/오프라인 표시용) |

### 관심사 분리 원칙

```
┌──────────────────────┐     ┌──────────────────────┐
│   TanStack Query     │     │      Zustand         │
│                      │     │                      │
│  - 재고 수량          │     │  - 장바구니           │
│  - 상품 정보          │     │  - UI 모달 상태       │
│  - 주문 상태          │     │  - 필터/정렬 조건     │
│  - 타임 세일 정보     │     │  - WebSocket 연결 상태│
│                      │     │                      │
│  서버가 원본인 데이터  │     │  클라이언트가 원본인   │
│                      │     │  데이터               │
└──────────────────────┘     └──────────────────────┘
```

## 5. 낙관적 업데이트 & 롤백

### 구매 플로우

타임 세일 환경에서 사용자 체감 속도를 높이기 위해 낙관적 업데이트(Optimistic Update)를 적용한다. 서버 응답을 기다리지 않고 즉시 UI를 갱신한 뒤, 실패 시 이전 상태로 롤백한다.

```
사용자 [구매 클릭]
       │
       ▼
  ┌─ onMutate ─────────────────────────┐
  │  1. 현재 캐시 스냅샷 저장            │
  │  2. 캐시의 재고 수량 즉시 차감       │
  │  3. 구매 버튼 비활성화               │
  └────────────────────────────────────┘
       │
       ▼
  서버로 구매 요청 전송 (Supabase RPC)
       │
       ├── 성공 ──▶ onSettled
       │            1. 서버 응답으로 캐시 최종 갱신
       │            2. 성공 피드백 표시
       │
       └── 실패 ──▶ onError
                    1. 스냅샷으로 캐시 롤백
                    2. 실패 사유 토스트 표시
                    3. 구매 버튼 재활성화
```

### TanStack Query Mutation 콜백 구조

| 콜백 | 역할 |
|------|------|
| `onMutate` | 이전 캐시 스냅샷 저장, 캐시 즉시 갱신 (낙관적 업데이트) |
| `onError` | 스냅샷으로 캐시 복원 (롤백), 에러 메시지 표시 |
| `onSettled` | 성공/실패 무관하게 최종 정리, 필요 시 쿼리 무효화로 서버와 동기화 |

### 롤백이 필요한 실패 사례

| 실패 유형 | 서버 응답 | 사용자 피드백 |
|-----------|-----------|---------------|
| 재고 부족 | `INSUFFICIENT_STOCK` | "죄송합니다. 해당 상품이 품절되었습니다." |
| 인당 수량 초과 | `PURCHASE_LIMIT_EXCEEDED` | "1인당 구매 가능 수량을 초과했습니다." |
| 세일 종료 | `SALE_ENDED` | "타임 세일이 종료되었습니다." |
| 네트워크 오류 | 응답 없음 | "네트워크 오류가 발생했습니다. 다시 시도해주세요." |

## 6. 이벤트 스로틀링 & 배칭

### 문제 상황

타임 세일 시작 직후, 재고 변경 이벤트가 초당 수십~수백 건 발생할 수 있다. 각 이벤트마다 즉시 리렌더링하면 프레임 드롭과 CPU 과부하가 발생한다.

### 스로틀링 전략

이벤트 수신과 UI 반영 사이에 스로틀링 계층을 둔다.

```
Realtime 이벤트 수신 (초당 N건)
       │
       ▼
  ┌─ 이벤트 버퍼 ──────────────────┐
  │  최신 상태만 유지 (덮어쓰기)     │
  │  productId 기준으로 중복 제거    │
  └────────────────────────────────┘
       │
       ▼
  requestAnimationFrame 콜백
  (약 16ms 간격, 60fps 기준)
       │
       ▼
  배치 캐시 업데이트
  (복수 상품의 변경을 한 번에 반영)
       │
       ▼
  단일 리렌더링 사이클
```

### 핵심 설계 원칙

| 원칙 | 설명 |
|------|------|
| 최종 상태 우선 | 동일 상품에 대한 연속 이벤트 중 마지막 값만 반영 |
| 프레임 동기화 | `requestAnimationFrame`으로 브라우저 렌더링 주기에 맞춰 업데이트 |
| 배치 처리 | 여러 상품의 변경 사항을 하나의 렌더링 사이클에서 일괄 처리 |
| 우선순위 분리 | 재고 0(품절) 이벤트는 스로틀링 없이 즉시 반영 |

## 7. 오프라인 복구 (Re-sync)

### 네트워크 단절 감지

| 감지 방법 | 설명 |
|-----------|------|
| WebSocket 연결 상태 | Supabase Realtime 채널의 연결 상태 콜백 활용 |
| `navigator.onLine` | 브라우저 네트워크 상태 API |
| Heartbeat 타임아웃 | WebSocket ping/pong 응답 지연 감지 |

### 복구 절차

```
네트워크 단절 감지
       │
       ▼
  Zustand: 연결 상태 → 'disconnected'
  UI: 오프라인 배너 표시
       │
       ▼
  자동 재연결 시도 (지수 백오프)
       │
       ▼
  연결 복구 시
       │
       ├── 1. Supabase Realtime 채널 재구독
       │
       ├── 2. TanStack Query 전체 무효화
       │      (stale 데이터 강제 refetch)
       │
       ├── 3. 서버 최신 상태로 캐시 갱신
       │
       └── 4. Zustand: 연결 상태 → 'connected'
            UI: 오프라인 배너 제거
```

### 오프라인 중 사용자 액션 처리

오프라인 상태에서 사용자가 구매를 시도하면, 요청을 차단하고 "네트워크 연결을 확인해주세요" 메시지를 표시한다. 재고의 정확성이 보장되지 않는 상태에서의 구매 시도는 허용하지 않는다.

## 8. 페르소나 관점 토론

### Monorepo Lead 의견

실시간 훅과 상태 로직의 모노레포 내 배치 전략은 다음과 같다.

| 패키지 | 위치 | 역할 |
|--------|------|------|
| `@repo/realtime` | `packages/realtime` | Supabase Realtime 구독 훅, 채널 관리, 이벤트 스로틀링 유틸리티 |
| `@repo/shared` | `packages/shared` | 공통 타입 정의, Supabase 클라이언트 설정, 쿼리 키 팩토리 |
| Web 앱 | `apps/web` | Web 전용 UI 컴포넌트, 페이지별 구독 설정 |
| Mobile 앱 | `apps/mobile` | 앱 전용 UI 컴포넌트, 푸시 알림 연동 |

- `@repo/realtime` 패키지에 공통 구독 훅(`useStockSubscription`, `useTimeSaleSubscription` 등)을 배치하여 Web과 App에서 동일한 로직을 공유한다.
- 플랫폼별 차이(예: 앱의 백그라운드 상태 처리)는 각 앱 프로젝트에서 래퍼 훅으로 확장한다.
- 쿼리 키 팩토리는 `@repo/shared`에 중앙 관리하여 키 불일치로 인한 캐시 오류를 방지한다.

### Engine Designer 의견

CDC 설정과 DB 성능에 대한 고려 사항은 다음과 같다.

**Publication 설정**

모든 테이블에 Realtime을 활성화하는 것은 성능 낭비이다. 실시간 동기화가 필요한 테이블만 선별적으로 Publication에 추가한다.

| 테이블 | Realtime 활성화 | 사유 |
|--------|:---:|------|
| `product_stock` | O | 핵심. 재고 변경을 실시간 브로드캐스트 |
| `time_sale` | O | 세일 상태(시작/종료) 변경 전파 필요 |
| `order` | O | 사용자 본인 주문 상태 변경 알림 |
| `product` | X | 상품 기본 정보는 변경 빈도 낮음, ISR로 충분 |
| `user` | X | 사용자 프로필 변경은 실시간 전파 불필요 |

**성능 고려 사항**

- WAL 수준의 CDC는 DB 쓰기 성능에 미미한 영향을 준다. 다만, 구독 클라이언트 수가 증가하면 Realtime 서버의 브로드캐스트 부하가 증가한다.
- 재고 테이블은 Row-Level 필터링을 적용하여, 클라이언트가 관심 있는 `productId`의 변경만 수신하도록 한다.
- 대량 트래픽 상황에서는 Supabase Realtime의 연결 수 제한을 모니터링해야 한다.

### Sync Master 의견

이 문서의 핵심 도메인으로서, 실시간 동기화 전략의 세부 사항을 다음과 같이 정리한다.

**구독 생명주기 관리**

- 컴포넌트 마운트 시 구독을 시작하고, 언마운트 시 반드시 구독을 해제한다. 구독 누수(Subscription Leak)는 메모리 누수와 불필요한 네트워크 트래픽의 원인이 된다.
- 동일 채널에 대한 중복 구독을 방지하기 위해, 구독 레지스트리를 유지하여 참조 카운팅 방식으로 관리한다.

**캐시 무효화 전략**

- Realtime 이벤트를 수신하면 `queryClient.setQueryData`로 캐시를 직접 갱신한다. `invalidateQueries`를 사용하면 추가 네트워크 요청이 발생하므로, 이벤트 페이로드에 충분한 데이터가 포함된 경우 직접 갱신을 우선한다.
- 다만, 이벤트 페이로드가 불완전하거나 데이터 정합성이 의심되는 경우(예: 오프라인 복구 직후)에는 `invalidateQueries`로 서버 데이터를 재요청한다.

**충돌 해결**

- 낙관적 업데이트와 Realtime 이벤트가 거의 동시에 도착하는 경우, 서버 데이터(Realtime 이벤트)를 항상 우선한다. 서버가 단일 진실 공급원이다.
- 낙관적 업데이트 중 Realtime 이벤트가 도착하면, 해당 mutation이 `onSettled`로 완료될 때까지 Realtime 이벤트 반영을 지연시키는 방법을 고려할 수 있다. 이를 통해 UI 깜빡임을 방지한다.

**에러 처리 패턴**

| 에러 유형 | 처리 방법 |
|-----------|-----------|
| 구독 연결 실패 | 지수 백오프로 재시도, 최대 5회 후 에러 표시 |
| 이벤트 파싱 오류 | 해당 이벤트 무시, 에러 로깅 |
| 캐시 갱신 실패 | 쿼리 무효화로 폴백 |
| 서버 상태 불일치 감지 | 전체 쿼리 무효화 및 재요청 |

### UI Artisan 의견

고빈도 업데이트 환경에서의 렌더링 전략은 다음과 같다.

**렌더링 최적화**

- 재고 수량 표시 컴포넌트는 `React.memo`로 감싸고, 재고 수량 값만을 의존성으로 설정하여 불필요한 리렌더링을 차단한다.
- 상품 목록에서 개별 상품의 재고 변경이 전체 목록을 리렌더링하지 않도록, 재고 표시를 독립 컴포넌트로 분리한다.

**애니메이션 전략**

| 상황 | 애니메이션 |
|------|-----------|
| 재고 감소 | 숫자 카운트다운 애니메이션 (이전 값 → 새 값 전환) |
| 품절 임박 (재고 5개 이하) | 재고 수량 강조 색상 변경 + 펄스 효과 |
| 품절 | "품절" 뱃지 페이드인, 구매 버튼 비활성화 전환 |
| 낙관적 업데이트 진행 중 | 구매 버튼에 로딩 스피너, 재고 수량에 일시적 스타일 변경 |
| 롤백 발생 | 재고 수량 복원 + 토스트 메시지 |

**시각적 피드백 원칙**

- 낙관적 업데이트가 적용된 값은 시각적으로 "확정 전" 상태임을 암시하되, 과도한 표현은 피한다 (예: 미세한 투명도 조정).
- 품절 전환은 사용자의 주의를 끌어야 하므로 즉시 반영하며, 스로틀링 대상에서 제외한다.

## 9. 구독 관리 생명주기

실시간 구독의 전체 생명주기를 단계별로 정리한다.

```
┌─────────────────────────────────────────────────────────────┐
│                    구독 생명주기                              │
│                                                              │
│  [1. Subscribe]                                              │
│   컴포넌트 마운트 시 Supabase Realtime 채널 구독 시작         │
│   구독 레지스트리에 등록 (참조 카운트 증가)                    │
│        │                                                     │
│        ▼                                                     │
│  [2. Listen]                                                 │
│   PostgreSQL CDC 이벤트 수신 대기                             │
│   이벤트 수신 시 콜백 실행                                    │
│        │                                                     │
│        ▼                                                     │
│  [3. Throttle]                                               │
│   이벤트 버퍼에 적재                                          │
│   동일 대상 이벤트 중복 제거 (최신 값만 유지)                  │
│   품절 이벤트는 즉시 통과                                     │
│        │                                                     │
│        ▼                                                     │
│  [4. Update]                                                 │
│   requestAnimationFrame 주기에 맞춰 배치 처리                 │
│   TanStack Query 캐시 직접 갱신 (setQueryData)               │
│   React 리렌더링 트리거                                      │
│        │                                                     │
│        ▼                                                     │
│  [5. Cleanup]                                                │
│   컴포넌트 언마운트 시 참조 카운트 감소                        │
│   참조 카운트가 0이 되면 채널 구독 해제                        │
│   이벤트 버퍼 초기화                                          │
└──────────────────────────────────────────────────────────────┘
```

### 단계별 책임 요약

| 단계 | 트리거 | 주요 동작 | 실패 시 대응 |
|------|--------|-----------|-------------|
| Subscribe | 컴포넌트 마운트 | 채널 생성, 이벤트 리스너 등록 | 재시도 (지수 백오프) |
| Listen | Realtime 이벤트 수신 | 콜백 실행, 이벤트 분류 | 파싱 오류 시 해당 이벤트 무시 |
| Throttle | 이벤트 버퍼 적재 | 중복 제거, 우선순위 판별 | 버퍼 오버플로 시 최신 값만 유지 |
| Update | requestAnimationFrame | 캐시 갱신, UI 반영 | 갱신 실패 시 쿼리 무효화 폴백 |
| Cleanup | 컴포넌트 언마운트 | 구독 해제, 리소스 정리 | 참조 카운트 불일치 시 강제 해제 |
