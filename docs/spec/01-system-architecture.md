## 2. 시스템 개요

moment-stock은 타임 세일 환경에서 초당 수천 건의 동시 구매 요청을 처리하면서 데이터 무결성을 보장하고, 모든 클라이언트에 0.1초 이내로 재고 상태를 전파하는 고성능 실시간 커머스 엔진이다.

시스템의 네 가지 핵심 기술 목표는 다음과 같다.

| 목표 | 설명 |
|------|------|
| **Zero-Lag Realtime Sync** | Supabase Realtime(CDC)을 활용하여 DB 재고 변화를 모든 클라이언트에 100ms 이내 전파 |
| **Atomic Concurrency Control** | PostgreSQL RPC 및 트랜잭션으로 동시 요청 시 초과 판매(Overselling) 가능성 0% 달성 |
| **60fps UI Optimization** | 고주파 데이터 업데이트 상황에서도 부드러운 UI를 유지하기 위한 렌더링 최적화 |
| **Unified Cross-Platform** | TurboRepo 모노레포를 통해 Web(Next.js)과 App(Expo) 간 비즈니스 로직 및 타입 공유 |

## 4. 계층별 역할 정의

### 4.1 Client Layer (클라이언트 계층)

| 구분 | 역할 |
|------|------|
| **Next.js 14 (Web)** | SSR/ISR을 통한 상품 정보 서빙, 재고 수량 실시간 스트리밍, 낙관적 UI 업데이트 및 롤백 |
| **Expo (App)** | 네이티브 푸시 알림 수신, 생체 인증 기반 간편 결제, 오프라인 복구 후 재동기화 |
| **공통 책임** | Supabase Realtime 채널 구독, TanStack Query를 통한 서버 상태 캐싱, Zustand를 통한 UI 전용 상태 관리 |

### 4.2 Shared Layer (공유 계층)

`@repo/shared` 패키지가 담당하며, TurboRepo를 통해 Web과 App이 동일한 코드를 참조한다.

- **Supabase Client 설정**: 환경별 클라이언트 인스턴스 팩토리
- **타입 정의**: 상품, 재고, 주문 등 도메인 모델의 TypeScript 타입
- **검증 스키마**: Zod 기반 입력 검증 스키마
- **비즈니스 유틸리티**: 재고 상태 판정, 할인율 계산 등 플랫폼 공통 로직

### 4.3 Backend Layer (백엔드 계층)

Supabase 인프라 위에 구축되며 다음 컴포넌트로 구성된다.

- **PostgreSQL**: 상품, 재고, 주문 데이터의 단일 진실 공급원(Single Source of Truth). CHECK 제약 조건과 RPC를 통한 원자적 재고 처리
- **Edge Functions**: 결제 검증, 재고 차감 트리거, 외부 서비스 연동 등 보안이 필요한 서버 사이드 로직
- **Auth**: JWT 기반 인증, Row Level Security(RLS) 정책과 연동하여 데이터 접근 제어
- **Storage**: 상품 이미지 등 정적 자산 관리

### 4.4 Realtime Layer (실시간 계층)

PostgreSQL의 Write-Ahead Log(WAL)를 감지하는 Change Data Capture(CDC) 방식으로 동작한다.

- **변경 감지**: 재고 테이블의 UPDATE 이벤트를 WAL에서 캡처
- **채널 브로드캐스트**: 변경 이벤트를 구독 중인 모든 클라이언트에 WebSocket으로 전파
- **클라이언트 수신**: 수신된 이벤트를 스로틀링/배칭하여 효율적으로 UI에 반영

## 5. 페르소나 관점 토론

### Monorepo Lead 의견

> "아키텍처의 성패는 패키지 경계를 얼마나 명확히 긋느냐에 달려 있다."

**구조 관점 분석:**

- `@repo/shared` 패키지는 Web과 App이 공유하는 **타입, 검증 스키마, Supabase 클라이언트 설정**만 포함해야 한다. UI 컴포넌트나 플랫폼 특화 코드가 유입되면 빌드 의존성이 꼬인다.
- TurboRepo의 `turbo.json`에서 빌드 파이프라인을 정의할 때, `@repo/shared`가 변경되면 Web과 App 모두 리빌드되도록 의존 그래프를 설정해야 한다.
- **순환 참조 금지 원칙**이 특히 중요하다. `shared -> web` 또는 `shared -> app` 방향의 의존이 발생하면 빌드가 불안정해진다. 의존 방향은 반드시 `web/app -> shared -> (외부 라이브러리)` 단방향이어야 한다.

**권고 사항:**

- 패키지를 `@repo/shared`, `@repo/ui`(공통 UI 프리미티브), `@repo/supabase`(DB 스키마 타입 자동 생성)로 분리하는 것을 검토할 것
- pnpm workspace의 `catalog:` 프로토콜을 활용하여 Supabase SDK, TanStack Query 등 핵심 의존성 버전을 단일 지점에서 관리할 것
- CI에서 `turbo run build --dry-run`을 통해 의존 그래프 변경을 감시할 것

### Engine Designer 의견

> "재고는 DB가 지킨다. 애플리케이션 계층에서 재고를 판단하는 순간 초과 판매가 시작된다."

**DB 아키텍처 관점 분석:**

- 재고 차감은 반드시 PostgreSQL RPC 함수 내에서 `UPDATE ... SET quantity = quantity - $1 WHERE product_id = $2 AND quantity >= $1` 형태의 원자적 쿼리로 수행해야 한다. 애플리케이션에서 SELECT 후 UPDATE하는 패턴은 Race Condition의 원인이 된다.
- `stock` 테이블에 `CHECK (quantity >= 0)` 제약 조건을 반드시 설정하여 DB 수준에서 음수 재고를 원천 차단해야 한다.
- RLS 정책 설계 시, 인증된 사용자만 구매 RPC를 호출할 수 있도록 하되, 재고 조회는 모든 클라이언트에 허용하여 실시간 전파와 충돌하지 않도록 해야 한다.

**권고 사항:**

- 타임 세일 상품 테이블과 일반 상품 테이블을 분리하여, 타임 세일 테이블에만 Realtime CDC를 활성화할 것. 불필요한 WAL 이벤트 발생을 억제하여 Realtime 채널의 부하를 줄인다.
- 주문 생성과 재고 차감을 하나의 트랜잭션으로 묶어 정합성을 보장할 것
- Edge Function에서 재고 차감 RPC 호출 전 요청 유효성 검증(중복 요청 방지, rate limiting)을 수행할 것

### Sync Master 의견

> "실시간이라는 말이 곧 모든 이벤트를 즉시 반영한다는 뜻은 아니다. 지능적으로 모아서 처리해야 한다."

**실시간 동기화 관점 분석:**

- Supabase Realtime CDC 이벤트를 클라이언트에서 수신할 때, 초당 수십~수백 건의 재고 변경이 발생할 수 있다. 매 이벤트마다 리렌더링하면 성능이 급격히 저하되므로, **이벤트 스로틀링**(예: 100ms 간격) 또는 **배칭**(다수 이벤트를 모아 한 번에 반영)이 필수적이다.
- TanStack Query의 `queryClient.setQueryData`를 통한 **낙관적 업데이트**가 핵심 흐름이다. 사용자가 구매 버튼을 누르는 즉시 UI에서 재고를 차감하고, 서버 응답이 실패하면 `onError` 콜백에서 이전 상태로 롤백한다.
- Zustand는 **UI 전용 상태**(타이머 카운트다운, 구매 버튼 활성화 여부 등)에만 사용하고, 서버 데이터(재고, 상품 정보)는 TanStack Query가 관리하도록 책임을 분리해야 한다.

**권고 사항:**

- Realtime 구독 시 `filter` 옵션을 활용하여 현재 화면에 표시 중인 상품의 재고 변경만 수신할 것. 전체 테이블 구독은 대역폭 낭비를 유발한다.
- 오프라인 복구 시나리오를 위해 TanStack Query의 `refetchOnReconnect` 옵션을 활성화하고, Realtime 채널 재구독 로직을 구현할 것
- 낙관적 업데이트의 롤백 시 사용자에게 명확한 피드백(토스트 알림 등)을 제공할 것

### UI Artisan 의견

> "재고가 줄어드는 것이 눈에 보여야 구매 욕구가 생긴다. 숫자만 바뀌는 것으로는 부족하다."

**UI/UX 관점 분석:**

- 타임 세일의 핵심 UX는 **시각적 긴박감**이다. 재고 수량이 줄어들 때 숫자 변경 애니메이션, 프로그레스 바 감소, 색상 변화(녹색 -> 주황 -> 빨강) 등 시각적 피드백이 구매 전환율에 직접 영향을 준다.
- Next.js의 App Router에서 상품 상세 페이지는 ISR로 정적 서빙하되, 재고 수량 영역만 클라이언트 컴포넌트로 분리하여 실시간 스트리밍해야 한다. 이 구조가 성능과 실시간성을 모두 확보하는 핵심이다.
- Expo에서 Tamagui를 사용하여 Web과 유사한 디자인 시스템을 구축하되, 네이티브 플랫폼의 특성(햅틱 피드백, 네이티브 스크롤 등)을 살려야 한다.

**권고 사항:**

- 재고 수량 컴포넌트를 원자적으로 분리하여 해당 컴포넌트만 리렌더링되도록 `React.memo`와 선택적 구독을 적용할 것
- 접근성(a11y)을 위해 재고 변경 시 `aria-live="polite"` 속성을 활용하여 스크린 리더 사용자에게도 재고 상태를 전달할 것
- 타이머 카운트다운은 `requestAnimationFrame` 기반으로 구현하여 탭 비활성화 시 불필요한 연산을 방지할 것
- Tailwind CSS의 `animate-pulse`, `transition-colors` 등을 활용하여 재고 임박 상태의 시각적 강조를 구현할 것

## 6. 핵심 설계 원칙

페르소나 토론을 종합하여 도출된 시스템 전반의 설계 원칙은 다음과 같다.

| 번호 | 원칙 | 설명 |
|------|------|------|
| 1 | **DB가 진실의 원천** | 재고 판단과 차감은 반드시 PostgreSQL 내부에서 원자적으로 수행한다. 애플리케이션 계층에서 재고를 판단하지 않는다. |
| 2 | **단방향 의존** | 패키지 의존 방향은 `Client -> Shared -> External`로 고정한다. 역방향 참조와 순환 참조를 금지한다. |
| 3 | **서버 상태와 UI 상태 분리** | 서버 데이터는 TanStack Query가, UI 전용 상태는 Zustand가 관리한다. 두 영역을 혼합하지 않는다. |
| 4 | **지능적 실시간 처리** | 모든 Realtime 이벤트를 즉시 반영하지 않는다. 스로틀링과 배칭으로 효율적으로 처리한다. |
| 5 | **낙관적 업데이트 + 확정적 롤백** | 사용자 체감 속도를 위해 낙관적으로 UI를 갱신하되, 실패 시 명확하게 롤백하고 피드백을 제공한다. |
| 6 | **렌더링 경계 최소화** | 재고 수량 등 고주파 갱신 영역은 독립 컴포넌트로 분리하여 리렌더링 범위를 최소화한다. |
| 7 | **플랫폼 특성 존중** | 공통 로직은 공유하되, 각 플랫폼(Web/App)의 고유 강점(SSR, 네이티브 기능)은 독립적으로 활용한다. |

## 7. 주요 데이터 흐름

### 7.1 상품 조회 흐름

```
[사용자] --요청--> [Next.js ISR / Expo 화면 진입]
                        |
                        v
               [TanStack Query: 캐시 확인]
                   |            |
              캐시 Hit       캐시 Miss
                   |            |
                   v            v
              캐시 반환    [Supabase PostgreSQL: 상품 + 재고 SELECT]
                                |
                                v
                        [TanStack Query: 캐시 저장]
                                |
                                v
                 [Supabase Realtime: 재고 채널 구독 시작]
                                |
                                v
                        [UI 렌더링 완료]
```

### 7.2 구매(재고 차감) 흐름

```
[사용자: 구매 버튼 클릭]
        |
        v
[낙관적 업데이트: UI 재고 즉시 차감 + 버튼 비활성화]
        |
        v

        |
        +--성공--> [주문 생성 확정] --> [UI 구매 완료 표시]
        |
        +--실패--> [TanStack Query 롤백: 재고 복원] --> [사용자 피드백 표시]
```

### 7.3 실시간 재고 업데이트 흐름

```
[다른 사용자의 구매로 인한 재고 변경]
        |
        v
[PostgreSQL: stock 테이블 UPDATE 실행]
        |
        v
[WAL: 변경 레코드 기록]
        |
        v
[Supabase Realtime CDC: 변경 이벤트 감지]
```
